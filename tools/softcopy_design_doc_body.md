# 1. 软件概述
TinyPKI 是一个面向物联网设备、边缘节点和弱网环境的轻量级公开密钥基础设施软件。软件以 C11 为实现语言，以 OpenSSL EVP 作为密码学运行后端，围绕 SM2、SM3、SM4 实现证书签发、吊销状态验证、身份认证与安全会话建立等核心能力。系统并不是把若干密码学原语简单拼接，而是围绕“轻量证书、离线吊销、认证即加密”三条主线组织为一条完整安全业务链路。

传统基于 X.509 证书、CRL 列表和 OCSP 在线查询的方案，在通用互联网环境中具有成熟生态，但当系统被部署到资源受限设备、边缘网关、工业控制节点或低带宽无线链路时，这些方案会暴露出明显问题。首先，显式证书体积较大，弱网传输成本高；其次，传统吊销校验普遍依赖第三方在线查询，网络抖动时会直接影响认证可用性；再次，业务层经常把证书校验、吊销校验和会话建立拆开处理，从而引入安全边界不一致的问题。TinyPKI 的设计目标，就是在不牺牲可验证性的前提下，将这些环节尽可能收束为一条轻量、稳定、默认安全的工程链路。

从应用定位看，TinyPKI 适用于以下典型场景：面向终端接入认证的边缘网关；要求低通信负担和本地状态判断的工业控制节点；无法持续访问互联网但仍需维持可信认证的局域设备组网；需要在微控制器或轻量服务组件内嵌入身份和会话安全能力的系统。软件既可部署为内存态 CA/RA 与边缘服务组件，也可裁剪后用于受限终端。

TABLE: 软件定位与应用范围
| 项目 | 内容 | 说明 |

| 实现语言 | C11 | 便于裁剪、移植、审计和嵌入 |

TinyPKI 的总体设计目标包括：第一，显著降低证书和认证前台消息的体积；第二，将吊销校验从传统在线查询改造为携带式离线验证；第三，在认证通过后直接建立受保护的会话，而不是把认证与加密割裂开；第四，为边缘多节点状态维护提供可扩展的同步与版本刷新能力；第五，通过高层 API 与默认拒绝策略降低业务层误用风险。

# 2. 系统总体设计
TinyPKI 的总体架构可以抽象为三层。第一层是“信任与签发层”，负责设备注册、身份授权、隐式证书签发以及根记录发布；第二层是“认证与吊销验证层”，负责在认证请求中组合证书、签名和非吊销证明，并由验证方基于本地缓存根记录完成离线判断；第三层是“会话与状态维护层”，负责在认证通过后建立安全会话，并在后台传播撤销状态和根记录版本。

从系统角色上看，TinyPKI 至少包含服务端与客户端两类主体。服务端负责模拟 CA/RA 与根记录发布者，执行设备身份注册、签发授权、撤销维护和根记录刷新；客户端负责导入签发结果、缓存根记录、导出认证材料、发起认证请求和建立安全会话。在更复杂的边缘部署场景中，还可以引入多个边缘节点共同参与撤销状态同步与版本一致性判断。

从工程组织上看，TinyPKI 的设计强调“高层流程封装”而非“底层原语拼装”。也就是说，调用者并不需要直接面对离散的椭圆曲线运算、哈希运算和多种状态结构，而是通过 service/client API 走完一条规范化业务主链。这样做的目的，是让可信 CA、证书用途、非吊销证明、握手绑定和会话派生在接口层面强绑定，从而把默认安全边界嵌入到软件结构本身。

TABLE: 总体架构分层
| 层次 | 主要职责 | 典型对象 |
| 信任与签发层 | 注册身份、授权签发、根记录发布 | PKI service、CA/RA 状态 |
| 认证与吊销层 | 校验证书、验证非吊销证明、检查用途 | PKI client、auth、revocation 模块 |
| 会话与维护层 | 派生会话密钥、保护业务数据、刷新状态 | AEAD 会话、delta/heartbeat/quorum |

# 3. 核心机制设计
TinyPKI 的技术价值主要不是某一个局部函数更快，而是整体链路的组织方式更适合受限环境。本章重点说明两条核心机制：其一是基于 ECQV 的轻量证书链路，其二是基于 Merkle 树的携带式非吊销证明机制。其中，后者是 TinyPKI 与传统 CRL/OCSP 路线差异最明显、也是系统设计中最具辨识度的部分。

## 3.1 基于 ECQV 的轻量证书链路
在证书设计上，TinyPKI 采用 ECQV 隐式证书，而不是直接沿用传统显式证书结构。终端先生成证书请求与临时材料，服务端在完成身份授权后签发紧凑结果，终端导入后在本地重构正式密钥对。该设计避免了在证书载荷中重复存放显式公钥和较重的签名结构，能够显著减小证书本体大小。

从系统效果上看，隐式证书并不仅仅意味着“文件更小”，更重要的是它把身份材料压缩到更适合弱网链路和受限存储的形式，使后续认证消息也获得了更好的上限。这为后面所有吊销证明、握手绑定和会话建立机制提供了一个更轻的载体基础。

## 3.2 基于 Merkle 树的携带式非吊销证明
TinyPKI 在吊销路径上的核心创新，是把传统“验证方主动在线查询状态”改造成“持证方主动携带非吊销证明”。这一改变不是简单地把 CRL 缩小一点，也不是把 OCSP 响应改短一点，而是改变了认证时的角色分工：发送方负责举证自己当前未被吊销，验证方负责在本地判断这份证据是否可信。

Merkle 树机制在这里承担两个作用。第一，它把撤销集合压缩为一个由根哈希代表的可验证结构，服务端只需发布带有 CA 签名的根记录，就能把当前撤销状态锚定下来。第二，它允许针对某个证书序列号构造紧凑的成员证明或缺席证明。对于正常未被吊销的证书，系统通过缺席证明来说明该证书序列号位于某两个已知位置之间，且不属于当前撤销集合。

这种设计的优点在于：验证方不必在认证时向外部再发一次网络查询，也不必下载一整份大体量撤销列表，而只需验证本地缓存根记录的可信性，再根据对方携带的证明重建哈希路径并完成判断。只要根记录已被本地接受，整个吊销校验过程就可以完全离线完成。

从工程角度看，Merkle 树路线不仅适合“验证是否被吊销”，还天然适合“版本化发布与多节点传播”。服务端可以围绕同一棵树持续生成新的根记录和差量更新，边缘节点则可以围绕版本号和根记录进行后台同步。这样，前台认证只承担必要的最小证明，而后台则维护全局状态的一致性。

TABLE: CRL、OCSP 与 TinyPKI 吊销校验思路对比
| 方案 | 前台动作 | 对网络的依赖 | 通信特征 | 系统特征 |
| 仅 CRL | 下载列表并本地查找 | 依赖列表可达 | 单次列表大、可缓存复用 | 更像批量同步 |
| OCSP + CRL | 在线逐条查询，失败时回退 CRL | 强依赖查询端点 | 单次响应小、请求频繁 | 更像在线问答 |
| TinyPKI | 发送方携带非吊销证明，接收方本地验证 | 主认证链路可离线完成 | 前台消息紧凑、无传统在线查询 | 更适合弱网和边缘场景 |

## 3.3 根记录、缓存与后台维护
TinyPKI 中的根记录并不是单纯的哈希值，而是一个被 CA 签名、带版本信息的权威状态对象。客户端和边缘节点可以在后台刷新它，认证阶段则只使用当前已缓存的版本来完成判断。这样的设计把“根状态传播”与“前台认证校验”解耦，使系统既具备离线认证能力，又保留了后续演进为更复杂边缘同步机制的空间。

需要强调的是，TinyPKI 的核心优势并不依赖复杂的前台缓存协商。也就是说，当前默认紧凑路径已经足够轻，后台缓存机制的意义主要在于帮助高复用节点降低重复通信，而不是给每次认证增加新的步骤。这一点对于弱网优化非常关键，因为前台额外一步带来的时延损失，往往会抵消节省若干字节的收益。

# 4. 业务流程与接口设计
TinyPKI 的业务流程可以分为冷启动和热路径两种情况。冷启动是指验证方尚未持有相关根记录或上下文缓存，此时发送方会提供较完整的认证材料；热路径是指验证方已经持有根记录和相关状态，此时发送方只需发送紧凑认证消息，从而减少重复通信。

## 4.1 冷启动认证流程
在冷启动场景中，终端首先准备证书、签名、根记录相关材料及非吊销证明，形成完整认证请求。验证方收到请求后，先验证信任根与证书合法性，再校验非吊销证明，最后校验握手绑定并派生会话密钥。该流程虽然比热路径稍重，但仍然不依赖传统在线 CRL 或 OCSP 查询，因此在受限网络环境下仍具有稳定性优势。

## 4.2 热路径认证流程
在热路径场景中，验证方已经缓存根记录，发送方只需发送紧凑认证消息和必要的证明即可继续认证。测试表明，这种方式能够进一步降低前台消息大小。热路径的设计目标不是盲目省掉所有字段，而是在不引入前台额外交互的前提下减少重复发送，从而使系统在重复通信场景中表现得更好。

## 4.3 接口与默认安全策略
TinyPKI 的高层 API 将服务端、客户端、证书、撤销、认证和会话保护能力封装为较稳定的调用面。系统默认要求可信 CA、根记录、用途约束和握手绑定同时满足时才允许继续认证，并在认证通过后直接建立受保护的安全会话。这种接口设计有助于降低业务系统由于拼装底层原语而带来的误用风险。

TABLE: 关键安全约束
| 约束项 | 作用 | 设计意义 |
| fail-closed 默认拒绝 | 缺少关键材料时拒绝继续 | 防止业务层绕过关键校验 |
| authority 绑定 | 限制根记录与信任主体匹配 | 防止跨 authority 混用 |
| key usage 检查 | 确保证书用途符合当前操作 | 防止授权边界被扩大 |
| 握手绑定 | 绑定临时密钥与认证上下文 | 防止身份与会话脱节 |
| 版本刷新 | 跟踪根记录与状态变更 | 为同步与回滚保护提供基础 |

# 5. 测试结果与方案对比
TinyPKI 的测试不是只针对单个函数耗时，而是围绕真实协议对象、受控本地交互和工作负载累计成本三种口径展开。这样做的目的是让结果既能说明本系统的核心优势，也能在与传统 CRL、OCSP 路线比较时保持较好的解释力。

## 5.1 单次对象与单次交互优势
从单次对象大小来看，ECQV 隐式证书为 79 字节，对照 X.509 DER 证书为 691 字节，隐式证书仅为后者的 11.43%。从单次前台交互看，TinyPKI 紧凑 HTTP 路径总字节为 560 字节，显著低于 OCSP 的 1537 字节，也远低于 CRL 的 29162 字节。这说明 TinyPKI 的优势并不是停留在理论结构上，而是已经直接反映在“每发起一次认证需要传多少数据”这一最核心的工程指标上。

TABLE: 关键结果摘要
| 指标 | 数值 | 结果含义 |
| X.509 DER 证书大小 | 691 B | 传统显式证书基线 |
| ECQV 隐式证书大小 | 79 B | TinyPKI 证书载荷明显更小 |
| 紧凑认证消息大小 | 214 B | 热路径前台消息较紧凑 |
| TinyPKI 紧凑 HTTP 总字节 | 560 B | 明显低于 OCSP 与 CRL |
| OCSP HTTP 总字节 | 1537 B | 小响应在线查询基线 |
| CRL HTTP 总字节 | 29162 B | 列表分发路径前台最重 |

## 5.2 累计工作负载结果分析
在 1000 次访问的 Zipf 场景下，TinyPKI 紧凑路径的累计总字节约为 583272.63 字节，已经接近 CRLite 建模基线的 580000 字节，并明显低于 OCSP+CRL 和仅 CRL 路线。更重要的是，TinyPKI 达到这一通信规模时并没有依赖传统在线状态查询，而是依靠本地根记录和携带式证明完成认证。

测试还表明，在 20 kbps 的弱网条件下，累计传输时延明显高于本地计算时延。这意味着 TinyPKI 当前的优化方向是正确的：最值得优先压缩的是前台通信负担，而不是单纯追求本地毫秒级计算进一步缩短。换句话说，TinyPKI 把主要优化资源投向了真正主导弱网体验的部分。

TABLE: 统一工作负载下的方案对比
| 方案 | 总字节 | 在线查询依赖 | 特点概括 |
| 仅 CRL | 8409154.32 B | 有 | 列表路径重，累计通信负担最高 |
| OCSP + CRL | 828465.77 B | 有 | 字节较 CRL 小，但需要频繁在线查询 |
| CRLite | 580000.00 B | 无传统在线查询 | 以前置更新换前台轻量 |
| TinyPKI 紧凑 | 583272.63 B | 无传统在线查询 | 累计通信量接近 CRLite，且认证链路更完整 |

从结果看，TinyPKI 并不试图在所有抽象层面都变成“最小开销”，而是在保持完整认证和会话建立能力的前提下，把前台通信成本压到接近 CRLite 的量级。这种结果对于受限设备和边缘场景更具实际价值，也更能体现系统化设计优势。

# 6. 部署、维护与演进建议
在部署方式上，建议采用“服务端发布根记录、客户端缓存根记录、边缘节点辅助同步”的结构。这样可以把状态维护和认证前台校验解耦：后台负责根记录与状态传播，前台负责最小必要认证消息和本地验证。对于连接频繁、对端重复率高的场景，可以继续围绕热路径优化做扩展，但不应以增加前台协商步骤为代价。

在维护方式上，建议始终保留统一的构建、测试和 benchmark 入口，使设计文档、源码文档和实验结果能够同步演进。这样既方便软著和专利材料准备，也便于后续版本变更时重新生成正式文档。软件当前已经具备较清晰的工程结构和较完整的测试结果基础，后续若继续演进，可优先围绕高复用场景下的热路径优化、根记录版本传播效率以及边缘多节点一致性辅助机制展开。
